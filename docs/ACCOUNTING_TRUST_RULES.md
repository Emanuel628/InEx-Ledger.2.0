# Accounting Trust Rules

This document is the single source of truth for how InEx Ledger handles mutation, locking, archiving, and exports. All routes, services, and UI elements must conform to these rules.

---

## 1. What Is Editable

| Field | Editable | Notes |
|---|---|---|
| Transaction amount | Yes | Creates an adjustment record (audit pivot) |
| Transaction date | Yes | Both original and new date must be outside the locked period |
| Category | Yes | Reassignment is allowed on unlocked transactions |
| Account | Yes | Reassignment is allowed on unlocked transactions |
| Description / note | Yes | Edit is allowed on unlocked transactions |
| Cleared status | Yes | Updates are allowed on unlocked transactions |
| Recurring template | Yes | Template-level fields only; does not rewrite past runs |
| Recurring template status | Yes | Pausing/activating does not affect already-posted transactions |

---

## 2. What Is Lockable

A **locked period** is a date range ending at `locked_through_date`, inclusive. Once set, no mutation can touch a transaction whose date falls on or before that date.

### Lock scope

| Operation | Locked period blocks it |
|---|---|
| Create transaction | Yes |
| Edit transaction (PUT) | Yes — both original date and new date are checked |
| Archive transaction (DELETE) | Yes |
| Cleared-status update (PATCH /cleared) | Yes |
| Recurring run materialization | Yes — occurrences dated inside the locked period are skipped |
| Manual recurring run (POST /recurring/:id/run) | Yes — returns 409 with `code: accounting_period_locked` |
| Receipt attachment (PATCH /receipts/:id/attach) | Yes — the transaction's date is checked |
| Category / account rename | No — name changes do not alter historical transaction rows |

### Lock behavior on error

All lock violations return HTTP **409** with the body:

```json
{
  "error": "Transactions dated on or before <date> are locked for this business.",
  "code": "accounting_period_locked",
  "locked_through_date": "YYYY-MM-DD"
}
```

### Indirect bypass prevention

- Recurring template edits that change `start_date` or `cadence` do not back-fill already-locked periods because `materializeTemplateRuns` skips any occurrence whose date is on or before `locked_through_date`.
- Category and account renames change display names only; no historical `category_id` / `account_id` foreign keys are altered.
- Batch routes (business data delete, privacy scrub) operate outside the lock because they act on the whole dataset under explicit user consent.

## 3. What "Delete" Means

**There is no hard delete for transactions.** "Delete" means **archive**.

When a user clicks "Archive" on a transaction:

1. `deleted_at` is set to the current timestamp.
2. `is_void` is set to `true`.
3. `voided_at` and `voided_by_id` are populated.
4. `deleted_by_id` and `deleted_reason` are populated.
5. The row is **retained** in the database indefinitely.

The action is **blocked** if the transaction's date is inside a locked period.

### What is preserved

All original fields remain intact. No data is overwritten. The archive metadata (`deleted_at`, `deleted_by_id`, `deleted_reason`, `is_void`, `voided_at`, `voided_by_id`) is appended.

### What "delete" does NOT mean

- It does not remove the row from the database.
- It does not remove the row from audit or history views.
- It does not remove associated receipts.

### Restore behavior

Restore is **not currently supported** through the UI. An archived transaction cannot be unarchived. This is intentional: if a transaction needs to be re-entered, create a new one.

### Who can archive

Any authenticated user who owns the business can archive a transaction, subject to the locked-period check.

---

## 4. Archive Behavior

| Requirement | Status |
|---|---|
| `deleted_at`, `deleted_by_id`, `deleted_reason` are always written | ✅ |
| `is_void`, `voided_at`, `voided_by_id` are always written | ✅ |
| Re-archiving an already-archived transaction is a no-op (returns null) | ✅ |
| Archived rows are excluded from all standard report queries | ✅ |
| Archived rows are excluded from analytics net-total calculations | ✅ |
| Archive is blocked by the locked-period check | ✅ |
| Archived rows are visible in GDPR / privacy data export | ✅ (by design — comprehensive data export) |

**SQL guard pattern** that must appear in every report / analytics query:

```sql
AND deleted_at IS NULL
AND (is_void = false OR is_void IS NULL)
AND is_adjustment = false
```

---

## 5. Reports and Exports

### Standard analytics

All analytics queries (dashboard, trends, category breakdown, budget vs actual) exclude:

- Archived transactions (`deleted_at IS NOT NULL` or `is_void = true`)
- Audit-adjustment rows (`is_adjustment = true`)

### PDF / CSV exports

PDF exports are generated by the pdf-worker service. The export covers the date range specified by the user. Archived transactions are excluded by the same SQL guards as analytics.

### Tax exports

Tax-mapped fields (category `tax_map_us`, `tax_map_ca`) apply only to live (non-archived, non-adjustment) transactions.

### GDPR data export

The privacy data export (`GET /api/privacy/data-export`) intentionally includes archived transactions as part of the complete personal data record. This is correct behavior under data-portability obligations.

---

## 6. What Users and CPAs Should Expect

| Scenario | Behavior |
|---|---|
| Editing a transaction in a locked period | Rejected with a clear "period is locked" error |
| Archiving a transaction in a locked period | Rejected with the same lock error |
| Viewing archived transactions | Not visible in the standard ledger view; visible in audit history |
| Running a report | Archived transactions are excluded from all totals |
| Exporting to PDF | Archived transactions are excluded |
| Recurring transaction auto-posting into a locked period | Occurrence is silently skipped; future occurrences continue normally |
| Manually running a recurring item into a locked period | Rejected with a 409 lock error |
